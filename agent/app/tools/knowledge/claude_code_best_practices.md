# Claude Code 最佳实践

## 核心原则：探索 → 规划 → 编码

不要一上来就写代码。遵循三步法：
1. **探索**：让 Claude 先阅读相关文件、理解现有架构
2. **规划**：用 /plan 模式制定方案，确认后再动手
3. **编码**：按计划逐步实现，每步验证

## 写好 Prompt 的技巧

### 提供具体上下文
- 指定文件路径：`查看 src/auth.py，重构 JWT 验证逻辑`
- 说明项目结构：`这是 FastAPI 项目，用 pytest 测试`
- 引用相关文件：`参考 tests/test_auth.py 的测试风格`

### 明确目标和约束
- 好的 prompt：`在 src/utils.py 中添加 email 验证函数，使用正则表达式，需要处理 Unicode 邮箱，添加单元测试到 tests/test_utils.py`
- 差的 prompt：`加个验证邮箱的功能`

### 拆分大任务
- 每次只做一件事：不要说"重构整个模块并写测试和文档"
- 用 /plan 先拆解：复杂任务让 Claude 先出方案再执行
- 迭代推进：完成一个小任务 → 验证 → 下一个

## CLAUDE.md 配置技巧

### CLAUDE.md 是什么
项目根目录的 CLAUDE.md 文件，Claude 每次对话都会自动读取。用于设定项目级别的规范和上下文。

### 写什么内容
- **项目概述**：技术栈、目录结构、关键约定
- **编码规范**：命名风格、错误处理模式、测试要求
- **架构约束**：不变量、禁止模式、模块职责边界
- **常用命令**：构建、测试、部署的具体命令

### 注意事项
- 保持简洁：过长反而干扰，聚焦真正有用的规则
- 具体明确：不要写"代码要优雅"，要写"所有公开方法必须有 docstring"
- 定期更新：项目变化后同步更新 CLAUDE.md

## 上下文管理

### 控制上下文窗口
- `/compact` — 压缩对话历史，释放 token 空间
- `/clear` — 清除历史，完全重新开始
- `/context` — 查看当前上下文使用量
- 新开会话比无限续对话效果更好

### 大文件处理
- 不要一次性粘贴整个大文件
- 用 `@文件路径` 引用文件，让 Claude 按需读取
- 指定关注范围：`只看 handle_request 函数`

## 常见失败模式

### 厨房水槽式对话
- 问题：一个会话塞太多不相关任务，上下文膨胀
- 解决：每个独立任务开新会话

### 反复纠正
- 问题：反复说"不对，再改"，Claude 越改越乱
- 解决：说清楚具体哪里错、期望是什么，而不是只说"不对"

### 过度信任
- 问题：不验证 Claude 生成的代码就直接用
- 解决：每次改动后运行测试，用 /diff 检查实际改了什么

### CLAUDE.md 过长
- 问题：CLAUDE.md 写了 500 行规则，反而干扰理解
- 解决：精简到最关键的 20-30 条规则

## 自动化与批处理

### 非交互模式
```bash
# 单次执行
claude -p "为 src/api.py 添加类型注解"

# 管道输入
git diff | claude -p "review 这些改动"

# JSON 输出（集成其他工具）
claude -p "列出所有 TODO" --output-format json
```

### 多任务并行
- 用 `claude` 命令启动多个独立会话
- 每个会话处理不同文件/模块
- 适合大规模重构场景

## MCP 服务器扩展

MCP（Model Context Protocol）让 Claude Code 连接外部工具和数据源：
- `/mcp` 管理已配置的 MCP 服务器
- 可连接数据库、API、自定义工具
- 在 CLAUDE.md 或 settings.json 中配置
