# B118 Evidence: message_history 角色重建 — ToolCallPart/ToolReturnPart 结构恢复

## 变更文件

- `server/app/runtime_api.py`

## 变更摘要

### 删除
- `_event_text_for_message_history()` 辅助函数（将所有事件扁平化为纯文本）

### 重写
- `_build_agent_message_history_from_events()`：从扁平文本模式改为结构化重建模式
  - `user_intent` → `ModelRequest(UserPromptPart)`
  - `tool_step(running)` → `ModelResponse(ToolCallPart)`
  - `tool_step(done/error)` → `ModelRequest(ToolReturnPart)` 配对最近的 running
  - `question` → `ModelResponse(ToolCallPart ask_user)`
  - `answer` → `ModelRequest(ToolReturnPart ask_user)` 配对最近的 question
  - `result` → `ModelResponse(TextPart)`
  - `streaming_text` / `phase_change` 等噪声事件跳过

### 新增
- `_extract_tool_args_from_event()`：从事件 payload 中提取工具调用参数

### 修复（代码审查发现）
- 原始代码使用单一 `last_tc_id` 跟踪所有 pending tool call，导致 tool_step 和 question 交叉场景下配对串扰
- 修复为 `pending_tool_step_id` 和 `pending_question_id` 两个独立变量分别追踪

## Code Review 发现

1. **[Fixed] tool_call_id 配对串扰**：原始修改使用单一 `last_tc_id` 变量，当 tool_step(running) 后出现 question 时，`last_tc_id` 被 question 覆盖，导致后续 tool_step(done) 配对到 question 的 id 而非 tool_step(running) 的 id。已修复为 `pending_tool_step_id` 和 `pending_question_id` 两个独立变量。

2. **[OK] Pydantic AI 规范兼容**：`ToolCallPart(tool_name, args, tool_call_id)` 和 `ToolReturnPart(tool_name, content, tool_call_id)` 构造参数与 pydantic_ai.messages 模块签名一致。

3. **[OK] ModelRequest/ModelResponse 交替**：重建后的结构满足 ModelRequest → ModelResponse → ModelRequest → ... 交替要求。
