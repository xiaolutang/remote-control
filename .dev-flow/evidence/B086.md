## Execution Evidence

Implemented terminal-scoped Agent session APIs and conversation event writes:

- Added terminal-scoped endpoints:
  - `POST /runtime/devices/{device_id}/terminals/{terminal_id}/assistant/agent/run`
  - `POST /runtime/devices/{device_id}/terminals/{terminal_id}/assistant/agent/{session_id}/respond`
  - `POST /runtime/devices/{device_id}/terminals/{terminal_id}/assistant/agent/{session_id}/cancel`
  - `GET /runtime/devices/{device_id}/terminals/{terminal_id}/assistant/agent/{session_id}/resume`
- Old non-terminal Agent run/respond/cancel/resume endpoints now return stable `400 terminal_id_required` instead of silently creating global Agent sessions.
- Runtime API validates `user_id + device_id + terminal_id + conversation_id + session_id` before respond/cancel/resume.
- Run creates or reuses the server-authoritative terminal conversation and treats client `conversation_id` only as compatibility metadata.
- Run records `user_intent` with `client_event_id` and `session_id`; duplicate run `client_event_id` reuses the active session when recoverable and does not start a second Agent run.
- Agent session manager now stores `terminal_id`, `conversation_id`, and `pending_question_id`.
- Agent question events include stable `question_id`.
- Agent trace/question/result/error/cancel/timeout events are persisted to `agent_conversation_events` before SSE emission for terminal-bound sessions.
- Respond requires `question_id + client_event_id`, records `answer`, wakes only the matching pending question, and maps duplicate answers to `409 question_already_answered`.
- Resume/cancel reject sessions that do not belong to the requested terminal conversation with `404`.

## Acceptance Check

- `run` ignores client-provided `conversation_id` as authority and binds to server conversation by `user_id + device_id + terminal_id`.
- `user_intent`, `question`, `answer`, `trace`, `result`, and `error` events are written to conversation events.
- `respond`, `resume`, and `cancel` use terminal-scoped paths and validate full session ownership.
- Resume replays the same terminal-bound session event cache after terminal ownership validation.
- Old clients missing `terminal_id` receive stable `terminal_id_required`.
- Respond requires `question_id + client_event_id`; duplicate `client_event_id` can return the existing answer event; duplicate answered question returns `409`.
