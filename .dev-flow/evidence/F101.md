## Execution Evidence

Implemented server-authoritative conversation projection on the client:

- `AgentSessionService` now fetches terminal-scoped conversation projection from `/api/runtime/devices/{device_id}/terminals/{terminal_id}/assistant/conversation`.
- `SmartTerminalSidePanelContent` loads that projection when the current `device_id + terminal_id` scope changes, rebuilds `_agentHistory` / `_traces` / `_currentQuestion` / `_agentResult` / `_agentError` as render cache, and no longer treats locally stitched prompt history as authority.
- Client-side `conversation_id` is now mutable and server-authoritative: it is populated from fetch projection or `session_created`, then reused on later `run`.
- If projection reports `active_session_id`, the panel automatically calls terminal-scoped `resume` after hydration so reopened mobile/desktop views recover the current Agent question or in-flight session.
- Terminal switching clears local Agent/planner cache for the previous terminal before loading the new terminal projection, preventing cross-terminal bleed.
- Added dedicated projection model parsing and extended event parsing to preserve `conversation_id`, `terminal_id`, and `question_id`.
- Added widget/unit coverage for projection fetch parsing, history hydration, active question restore, terminal switch isolation, and server conversation id reuse.

## Acceptance Check

- Opening the smart panel hydrates the current terminal conversation from server events and renders the same history.
- `run/respond/resume` operate on `device_id + terminal_id`, and subsequent `run` uses the server-returned `conversation_id`.
- Local `_agentHistory` is render cache only; prompt authority no longer comes from `_buildAgentPrompt`.
- Empty server projection yields an empty panel instead of reusing another terminal's local history.
- Switching terminals clears prior cache and replaces it with the newly fetched terminal projection.
