## Execution Evidence

Implemented live client-side synchronization for the terminal-bound Agent conversation:

- Added `AgentSessionService.streamConversation(...)` for `/api/runtime/devices/{device_id}/terminals/{terminal_id}/assistant/conversation/stream?after_index=N`, parsing `conversation_event` SSE frames into `AgentConversationEventItem`.
- `SmartTerminalSidePanelContent` now keeps a server event cache plus `next_event_index`, so after the initial projection fetch it can replay remote delta events into the same render cache used by desktop/mobile.
- When there is no `active_session_id`, the panel subscribes to the terminal conversation stream and applies remote `user_intent/question/answer/result/error` events live.
- While a local/active Agent session is in progress, the panel still uses run/resume SSE as the primary channel; after result/error/cancel it resumes the conversation stream so subsequent turns from the other device stay in sync.
- Added terminal-closed handling on the panel: a streamed `closed` event or local terminal status closure clears the smart conversation UI, shows a closed-state message, and disables the smart input box/send button.
- Input submission guards now refuse `run/respond/planner` actions once the terminal conversation is marked closed, preventing smart input from conflicting with a closed terminal lifecycle.

## Acceptance Check

- If one device starts an Agent conversation, the other device can see the same conversation state through fetch + stream synchronization.
- Remote answers and subsequent results update the current panel without requiring local prompt stitching.
- `terminal closed` transitions the smart panel into a closed, non-interactive state.
- Smart input remains isolated from the terminal input channel because only the side-panel submit path triggers Agent requests; terminal transport is unchanged.
- Mobile still has no local Agent tool runtime; synchronization is UI/network only.
