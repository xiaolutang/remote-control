## Execution Evidence

Implemented terminal conversation projection and incremental stream APIs:

- Added `GET /runtime/devices/{device_id}/terminals/{terminal_id}/assistant/conversation`.
- Added `GET /runtime/devices/{device_id}/terminals/{terminal_id}/assistant/conversation/stream?after_index=N`.
- Added `AgentConversationProjection` and `AgentConversationEventItem` response models.
- Fetch validates current user owns `device_id` and that `terminal_id` exists and is not closed.
- Fetch returns empty projection with `conversation_id=null`, `events=[]`, `next_event_index=0` when no active conversation exists.
- Fetch returns full event projection for active conversations, including `active_session_id` when an in-memory Agent session is still exploring or asking.
- Stream validates the same device/terminal ownership boundary and emits only events with `event_index > after_index`.
- Stream uses polling-backed SSE keepalive so clients can subscribe before another endpoint appends events.
- Added `AgentSessionManager.get_active_terminal_session(...)` to expose active session lookup without scanning manager internals from the API layer.

## Acceptance Check

- GET conversation returns the same terminal conversation metadata and event sequence.
- Stream with `after_index=N` only emits events after `N`.
- Events appended by run/respond are visible through fetch/stream because both read server `agent_conversation_events`.
- No active conversation returns an empty projection instead of fabricated local history.
- API uses authenticated user context and validates device/terminal ownership before returning projection or stream.
