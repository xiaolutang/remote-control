### 实现细节

**1. 信号处理器 — IntegrationRunner**

新增 `install_signal_handlers()` 和 `restore_signal_handlers()` 方法：
- 注册 SIGINT/SIGTERM 处理器
- 信号触发时调用 `tear_down()` 清理 Docker 资源
- 清理后抛出 `KeyboardInterrupt` 使进程退出
- 支持恢复原始信号处理器

**2. cleanup_old_runs — db.py**

`EvalDatabase.cleanup_old_runs(keep_last=10)`:
1. 查询所有已完成 run（completed_at IS NOT NULL），按 completed_at DESC 排序
2. 保留前 N 个，删除其余已完成的 run
3. 活跃 run（completed_at IS NULL）完全不受影响
4. 级联删除：grader_results → quality_metrics → trials → runs

**3. cleanup CLI 子命令**

```bash
python -m evals cleanup --keep-last 10
```

  - [代码质量-错误处理] `_cmd_cleanup` 缺少异常处理：若 `EvalDatabase(args.db)` 路径不可写或 `cleanup_old_runs` 执行中途失败（如磁盘满），没有 try/except 包裹，会以 traceback 退出而非友好错误信息。这是 low severity 问题，因为 CLI 工具的 traceback 对开发调试可接受，但生产化时应包装。
  - [代码质量-SQL] cleanup_old_runs 中使用 f-string 拼接 IN 子句的 placeholders（如 `f"... IN ({placeholders})"`），虽然 placeholders 本身是 `?` 占位符，不存在 SQL 注入风险，但 delete_ids 列表可能非常大（如用户积累了数千个 run），会导致 SQL 语句过长。SQLite 对单个 SQL 的变量数量限制为 999（默认 SQLITE_MAX_VARIABLE_NUMBER），超出会报错。建议在后续迭代中增加分批删除逻辑。Medium severity。
  - [安全性-信号处理] `install_signal_handlers` 实现合理：capture self 引用、保存原始处理器、在 handler 中先 tear_down 再恢复原始处理器再 raise KeyboardInterrupt。但有一个潜在问题：如果 `tear_down()` 本身抛出异常，后续的 `signal.signal` 恢复和 `raise KeyboardInterrupt()` 都不会执行，进程可能进入不确定状态。建议在 handler 内对 `tear_down()` 加 try/except 确保恢复逻辑总是执行。Low severity。
  - [安全性-信号处理] 信号处理器中同时恢复 SIGINT 和 SIGTERM 的原始处理器是正确的做法，避免了信号处理器在非信号场景下被调用时误恢复。通过。
  - [安全性-信号处理] `restore_signal_handlers` 使用 `hasattr` 检查是安全的，避免了未安装处理器时调用出错。通过。
  - [测试覆盖] 9 个测试覆盖了：SIGINT/SIGTERM 触发清理、恢复验证、保留最近 N、删除旧数据及关联、保护活跃 run、无数据可删、空 DB、CLI 集成。覆盖面充分。通过。
  - [测试覆盖-缺失] 缺少以下边界测试：(1) `--keep-last 0` 行为（应删除全部已完成 run，当前逻辑可正确处理）；(2) `--keep-last -1`（argparse type=int 会接受负数，cleanup_old_runs 的 LIMIT -1 在 SQLite 中行为未定义）；(3) tear_down 抛异常时信号处理器的行为。Low severity。
  - [代码质量-幂等性] cleanup_old_runs 是幂等的（多次调用结果一致），这是好的设计。通过。
