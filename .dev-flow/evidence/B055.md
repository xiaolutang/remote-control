### 4. 关键设计决策

- **无法计算的时间指标返回 -1.0**（而非 0.0），以便区分"无法计算"和"瞬间完成"
- **integration 事件由 harness 在循环中实时采集**，而非事后从 transcript 转换（确保时间戳准确）
- **source 过滤在 DB 层实现**（_build_metric_conditions），query 和 aggregate 共享

  - [安全性] source 参数通过参数化查询传入 DB（`conditions.append("source = ?"); params.append(source)`），无 SQL 注入风险。API 层 source 参数是 Optional[str] 直接透传，无校验，但值只用于 DB 过滤，不做命令拼接，安全。
  - [代码质量-中] harness.py 中 `integration_events` 的采集逻辑散布在 `_execute_single_trial` 的多处（phase_change/streaming_text/tool_step/result/error），每个工具类型（execute_command/ask_user/lookup_knowledge）各有一段几乎相同的 `integration_events.append(...)` 代码。这增加了维护成本。但考虑到 harness 本身的结构已经很长，且事件采集逻辑与业务逻辑耦合紧密，当前实现可接受。
  - [测试覆盖] 25 个新测试覆盖 5 个效率指标的边界情况：无事件/单事件/多事件/从事件流提取、source 过滤查询、integration source 设置、数据隔离。全部通过。
  - [效率] 5 个计算函数均为 O(1)（从 SessionEventData 字段直接计算），无性能问题。harness 中 `datetime.now(timezone.utc).isoformat()` 调用频繁（每个事件一次），但数量有限（每个 trial 几十次），不影响性能。
  - [数据一致性-注意] harness 中 result 事件的 usage 字段使用 `total_token_usage`（累加值），但 `completion_tokens` 和 `output_tokens` 都设为 `total_token_usage.get("output_tokens", 0)`，而 `total_token_usage` 在每轮 LLM 调用中累加 input_tokens/output_tokens/total_tokens。这意味着 result 事件的 usage 是整个 trial 的累加值而非单次 LLM 调用的值，与 quality_monitor 的设计一致（`extract_session_data` 从 result 事件提取 `data.total_tokens`）。
