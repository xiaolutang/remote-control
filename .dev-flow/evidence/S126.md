## 根因分析

### 问题 1: Grader 注册表名与 YAML 配置名不匹配

YAML 任务配置中使用了 10 种 grader 名称，但注册表中只有 6 种（5 code grader + 1 llm_judge）。

**不匹配对照表:**

| YAML 配置名 | 注册表名 | 状态 |
|---|---|---|
| `response_type_match` | `response_type_match` | 匹配 |
| `steps_structure` | `steps_structure` | 匹配 |
| `contains_command` | `contains_command` | 匹配 |
| `steps_contain_match` | (不存在) | **缺失** |
| `safety_rejection` | (不存在) | **缺失** |
| `knowledge_relevance` | (不存在) | **缺失** |
| `knowledge_miss_handling` | (不存在) | **缺失** |
| `step_append` | (不存在) | **缺失** |
| `context_reference` | (不存在) | **缺失** |
| `intent_correction` | (不存在) | **缺失** |

7 种 YAML 配置的 grader 根本没有注册，精心设计的评分逻辑完全未生效。

### 问题 2: `_evaluate_trial` 绕过 grader 注册表

`harness.py` 的 `_evaluate_trial` 方法内联实现了简单的 response_type + steps 子串匹配，完全绕过了 `task.graders` 配置和 `GRADER_REGISTRY`。

## 修复方案

### 1. 新增 7 个 grader 类

在 `server/evals/graders/code_grader.py` 中新增:

- `StepsContainMatchGrader` (grader_type: `steps_contain_match`)
- `SafetyRejectionGrader` (grader_type: `safety_rejection`)
- `KnowledgeRelevanceGrader` (grader_type: `knowledge_relevance`)
- `KnowledgeMissHandlingGrader` (grader_type: `knowledge_miss_handling`)
- `StepAppendGrader` (grader_type: `step_append`)
- `ContextReferenceGrader` (grader_type: `context_reference`)
- `IntentCorrectionGrader` (grader_type: `intent_correction`)
- `ContentQualityGrader` (grader_type: `content_quality`)

### 2. 修改 `_evaluate_trial` 使用 grader 注册表

- 当 `task.graders` 有实质性配置（非默认 `["exact_match"]`）时，调用注册表 grader
- 保留内置逻辑作为向后兼容回退

### 3. 添加别名机制

新增 `GRADER_ALIASES` 字典和修改 `get_grader` 支持别名查找，为未来兼容性预留扩展点。

## 修改文件

1. `server/evals/graders/code_grader.py` - 新增 8 个 grader + 别名机制
