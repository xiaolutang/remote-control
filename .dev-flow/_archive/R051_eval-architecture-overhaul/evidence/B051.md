## Acceptance Criteria
1. 同一 terminal 连续两次 agent run，第二次复用同一个 session_id（不创建新 session）
2. 不同 terminal 的 agent run 使用不同 session_id
3. 终端删除时关联的 agent session 被清理
4. 终端因设备离线、用户登出、认证失效等非删除原因关闭时，session 标记为 inactive
5. 同一终端重连或下次 run 时，inactive session 自动恢复为 active 并继续复用
6. agent 内部 run 概念保留
7. SSE 事件中 session_created 只在首次 run 时发送
8. usage_store 同一 session_id 多次累加正确
9. Redis 不可用时 session 查找返回 503
10. 首次 agent run LLM 失败时不创建 session 记录
11. 同一 terminal 多次 run 的 execution report 按 run_id 区分
12. GET /api/agent/usage/summary?terminal_id=yyy 返回 per-terminal 累计 usage
13. terminal 无 session 时 usage summary 返回零值
14. terminal_id 不属于该 device_id 时返回空零值
15. pytest 全部通过

### Implementation (2026-04-29)

**核心改动**：

1. **agent_session_types.py** — 新增 `INACTIVE` 状态（终端非删除关闭）

- #65: 服务器是 token usage 唯一权威来源
- 禁止创建新 agent session_id per question
- 禁止客户端自行累加 token usage

- **provider**: codex_plugin
- **status**: conditional_pass

- **findings**:
  - [架构一致性] session 与 terminal 1:1 不变量（#64）实现正确。`generate_terminal_session_id` 确定性派生 session_id，同一 terminal 始终映射同一 session_id，不依赖客户端传入。禁止模式"每次提问创建新的 agent session_id"已通过 `agent_api.py` 中删除 `uuid4().hex` 分支落实。不变量 #65（服务端 token 统计权威源）也通过 `usage_store` 的 `terminal_id` 维度查询保持。整体符合 architecture.md 要求。
  - [架构一致性] `INACTIVE` 状态的引入合理。终端非删除关闭（设备离线、用户登出、认证失效）与删除场景正确区分。`reuse_or_create_session` 对 INACTIVE/COMPLETED/ERROR/EXPIRED/CANCELLED 状态的 session 统一走重置逻辑，符合不变量 #64 的 lazy creation 语义。
  - [代码质量] `generate_terminal_session_id` 取 `terminal_id[:16]` 存在理论碰撞风险：两个不同 terminal_id 前 16 字符相同时会映射到同一 session_id。当前 terminal_id 是 UUID 格式（32 hex），前 16 字符碰撞概率极低，但未在代码中做防御性断言。建议添加注释说明此假设或增加 full-id fallback。
  - [代码质量] `reuse_or_create_session` 方法 60+ 行，职责多（查找+状态判断+重置+创建），但逻辑清晰、注释充分，可接受。`remove_terminal_sessions` 和 `mark_session_inactive` 简洁明了。
  - [代码质量] `AgentSession` 新增的 `run_count`, `is_first_run`, `current_run_id` 字段命名清晰，default 值合理。`current_run_id` 使用 `uuid4().hex` 作为每次 run 的唯一标识，可区分同一 session 的多次 run。
  - [错误处理] `mark_session_inactive` 对不存在 session_id 做 silent no-op 是合理的。`remove_terminal_sessions` 遍历 `list(self._sessions.items())` 避免迭代中修改字典，正确。
  - [安全性] `agent_usage_api.py` 的 `terminal_id` 查询参数未做归属校验——用户可以传入不属于自己 device 的 terminal_id 查询 usage。但 evidence 文件已承认此限制（AC#6），且 usage 查询本身按 `user_id` 过滤，不会泄露跨用户数据，风险可控。
  - [测试覆盖] 27 个新测试覆盖了核心场景：session 复用、terminal 隔离、终端删除清理、inactive/reactivate、run failure 隔离、concurrent run 防护、usage 累加、report 按 run_id 区分。测试质量较高。
  - [测试覆盖-不足] `test_session_not_persisted_on_llm_failure` 测试逻辑较弱——它只验证 mock_save_usage 未被调用，但没有模拟 LLM 调用失败的完整流程。AC#10 的覆盖不够深入。
  - [测试覆盖-不足] `test_result_event_contains_run_id` 是一个手动构造 dict 的断言测试，没有验证真实 result event 生成流程中是否包含 run_id，实际验证价值有限。
  - [效率] `get_terminal_session` 线性遍历 `_sessions` 字典（O(n)）。当前 session 数量有限，可接受。但如果 session 数增长，建议增加 `terminal_id → session_id` 的反向索引。
  - [效率] `reuse_or_create_session` 在重置 session 时清空 `event_queue`（`asyncio.Queue()`）、`_last_events`、`_stream_ref_count` 等，清理全面，无资源泄漏。
