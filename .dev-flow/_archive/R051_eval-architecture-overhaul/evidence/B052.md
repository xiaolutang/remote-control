- **findings**:
  - [架构一致性] feedback→eval 闭环设计符合 R044 评估体系约束：eval 框架在 `server/evals/` 独立包内，使用独立 `evals.db`，不侵入业务代码的表结构。`_trigger_quality_monitor` 通过函数内 import evals 模块，实现了延迟依赖，业务代码不直接耦合 eval 模块。
  - [架构一致性] `quality_monitor` 自动触发在 result 事件后执行，属于只读指标提取（符合 architecture.md "质量指标提取只读 agent_conversation_events 元数据"要求），不修改业务状态，不影响主流程。
  - [代码质量] `_trigger_quality_monitor` 函数设计得当：函数内 import 避免循环依赖，`asyncio.ensure_future` 实现真正的异步非阻塞，双层 try/except 确保触发失败不影响 agent loop。best-effort 语义明确。
  - [代码质量] `_run_analyze_feedback` 包装函数设计合理：独立 async 函数，`await eval_db.init_db()` 确保数据库就绪，异常捕获完整，日志级别正确（warning for unexpected，info for skip）。
  - [代码质量] `feedback_api.py` 的 `FeedbackCreateRequest` 将 `session_id` 从必填改为 Optional，向后兼容处理正确——旧客户端传 `session_id` 正常工作，新客户端可不传。`session_id or ""` 处理 None 值，不会传 None 到下游。
  - [代码质量-问题] `feedback_service.py` 中 `create_feedback` 内部 `asyncio.ensure_future(_run_analyze_feedback(...))` 的调用存在一个问题：`ensure_future` 不保证任务在当前 event loop 上执行（虽然实际上通常如此）。在 FastAPI 的 async handler 中使用 `ensure_future` 创建的后台任务不受请求生命周期管理，如果 server 正在 shutdown，该任务可能被丢弃。生产环境建议使用 `BackgroundTasks` 或 `asyncio.create_task` 配合显式引用。当前实现是可接受的最简方案。
  - [代码质量-问题] `_trigger_quality_monitor` 中 `from evals.db import EvalDatabase` 是函数内 import，每次 result 事件都会执行 import（虽然 Python 会缓存模块），且每次创建新的 `EvalDatabase` 实例。如果 result 事件频繁，建议缓存 eval_db 实例或使用全局单例。当前实现是 best-effort 场景，可接受。
  - [安全性] `feedback_api.py` 新增的 `feedback_type` 使用 `Literal["helpful", "needs_improvement", "error_report"]` 枚举约束，Pydantic 自动验证无效值返回 422。`terminal_id` 和 `result_event_id` 作为 Optional[str] 透传，无注入风险（下游 log-service 是 HTTP 调用，不直接拼 SQL）。
  - [安全性] feedback→eval 闭环中，`_run_analyze_feedback` 只传 `feedback_id`, `category`, `description` 到 analyze_feedback，不传原始 session_id 或 user_id，符合 architecture.md "反馈闭环只接收脱敏摘要"的约束。
  - [测试覆盖] 15 个新测试覆盖了关键场景：新字段提交+透传、旧格式兼容、部分字段、session_id 可选、feedback_type 验证、analyze 触发、analyze 失败降级、quality_metrics 新字段持久化、默认值、integration source、trigger 函数存在性、空事件不触发、有事件不抛异常。覆盖度充分。
  - [测试覆盖-不足] `test_create_feedback_triggers_analyze` 使用 `asyncio.sleep(0.05)` 等待 ensure_future 调度，这是非确定性测试——在慢 CI 环境下可能 flaky。建议使用 mock + 显式 assert 或 run_until_complete 模式。
  - [测试覆盖-不足] `TestQualityMonitorAutoTrigger` 中 `test_trigger_handles_events` 的 session 没有 `_last_events` 属性的默认初始化（`AgentSession` dataclass 中 `_last_events` 可能没有默认值），依赖 `AgentSession` 内部实现细节。如果 `_last_events` 字段名变更，测试会静默通过（因为没有事件就不触发），无法检测回归。
  - [效率] `extract_and_store_metrics` 签名扩展使用 keyword-only 参数（`*` 后），向后兼容，不影响既有调用方。`compute_all_metrics` 同理。扩展方式干净。
  - [数据库] `quality_metrics` 表 ALTER 使用 `ALTER TABLE` 加列 + DEFAULT 值，对已有数据安全。`INSERT OR REPLACE` 语句已更新包含新字段，backward-compatible（`row.get("source", "production")` 在 from_row 中有 fallback）。
