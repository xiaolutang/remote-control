## Session S017 - 电脑离线后的 terminal 收口与统一关闭语义
> 时间: 2026-03-28 23:58
> 类型: bugfix
> 状态: confirmed
> 来源: 真实使用反馈
> **标记: 缺陷回流**

### 变更类型
- **类型**: 状态机收敛 + 服务端清理语义修正 + 客户端展示修正
- **影响范围**: Server 设备离线后的 terminal 状态收口、Agent 断开后的 terminal 生命周期、Flutter workspace 在线/离线与 terminal 可用性展示
- **紧急程度**: 高（当前“电脑离线但 terminal 仍像活着”会直接影响创建逻辑和用户理解）

### 用户反馈（真实使用发现）
1. 能否创建 terminal 的前置条件只应看：`电脑在线 && 当前 terminal 数 < 服务端配置上限`。
2. 反过来理解，若电脑关闭或 Agent 离线，则这台电脑上的 terminal 也应一并关闭或至少全部变为不可用。
3. `max_terminals` 是服务端配置；普通后端接口不应接受客户端传入该参数。
4. 终端一旦被关闭，就不应再维持其连接记录，这些记录没有实际意义。

### 已确认结论
1. `电脑在线` 继续只等于：**本机 Agent 在线，且当前电脑可创建并承载 terminal**。
2. 设备离线后，这台电脑上的 terminal 不再被视为可运行 terminal；不允许继续 attach、输入或新建。
3. terminal 若因设备离线而失去承载条件，应统一收口为 `closed` 或等价的不可用终态，而不是继续保留为活动 terminal。
4. `closed terminal` 不再维持活动连接记录，也不参与活动 terminal 数、视图数和创建判断。
5. `max_terminals` 仍是服务端权威配置；客户端只读展示，普通 runtime API 不接受客户端修改。

### 规划结论
1. 新增“设备离线后的 terminal 收口基线”任务，明确 `device offline => terminals unavailable/closed` 的状态语义。
2. 新增“服务端离线即清理 terminal 活跃态”任务，统一 Agent 断开后的 terminal 状态收口、连接记录清理与名额释放。
3. 新增“客户端离线展示与 terminal 可用性对齐”任务，避免离线时仍显示可切换/可连接的 terminal。
4. 新增“首个 terminal 创建与离线恢复链回归”任务，验证 `offline -> 恢复 Agent -> create` 以及 `offline -> terminals closed` 不回归。

### 待确认项
- [x] 电脑离线后，这台电脑上的所有 terminal 都应关闭或变为不可用 ✅
- [x] `closed terminal` 不再维持活动连接记录 ✅
- [x] `max_terminals` 继续仅由服务端配置，普通接口不接受客户端传入 ✅

## Session S015 - Terminal 创建准入、Agent 启动与 closed 清理语义
> 时间: 2026-03-28 23:59
> 类型: bugfix
> 状态: confirmed
> 来源: 真实使用反馈
> **标记: 缺陷回流**

### 变更类型
- **类型**: 语义收敛 + 服务端准入逻辑修正 + 桌面端首用流程补丁
- **影响范围**: Server runtime API、Agent 启动语义、Flutter workspace 创建流、关闭 terminal 清理策略
- **紧急程度**: 高（已经影响新建 terminal、close 后重建与用户对在线条件的理解）

### 用户反馈（真实使用发现）
1. 能否创建 terminal 的条件应收敛为：`电脑在线 && 当前 terminal 数 < 服务端配置上限`。
2. 若 Agent 被关闭且当前 terminal 数为 0，桌面端应先恢复/启动本机 Agent，再创建第一个 terminal。
3. `max_terminals` 是服务端配置；普通后端接口不应再接受客户端传入该字段。
4. terminal 一旦被关闭，就不应再维持它的连接记录或让其继续参与活动状态计算。

### 已确认结论
1. `电脑在线` 仍只表示：**本机 Agent 在线，且当前电脑可创建并承载 terminal**。
2. `创建 terminal` 的前置准入条件只保留两条：
   - `agent_online = true`
   - `active_terminals < max_terminals`
3. Agent/PTY 拉起失败属于“创建执行失败”，不属于“允许创建”的定义本身。
4. 对桌面端首个 terminal 创建，若本机 Agent 离线且 terminal 数为 0，应先尝试恢复/启动本机 Agent，再继续创建。
5. `max_terminals` 改为服务端内部/管理员配置；普通 runtime device 更新接口不再接受该字段。
6. `closed terminal` 不再维持活动连接记录，不再参与视图数、活动 terminal 数和创建判断；只保留必要的轻量历史记录。

### 规划结论
1. 新增“terminal 创建与清理语义基线”任务，统一准入条件、Agent 启动前置和 closed terminal 清理定义。
2. 新增“服务端创建准入与 closed cleanup”任务，移除公开接口里的 `max_terminals` 入参，并让 closed terminal 彻底退出活动集合。
3. 新增“桌面端首个 terminal 创建前自动恢复 Agent”任务，避免本机桌面端出现“App 打开但第一个 terminal 无法建立”的断层体验。
4. 新增“workspace 创建/关闭文案与交互”任务，统一空态、失败提示和 close 后重新创建流程。

### 待确认项
- [x] 创建 terminal 的准入条件只看电脑在线和 terminal 数量上限 ✅
- [x] 本机桌面端在 Agent 离线且 terminal 为 0 时，先恢复 Agent 再创建 ✅
- [x] `max_terminals` 不再由普通 runtime API 接受客户端修改 ✅
- [x] `closed terminal` 不再维持连接记录或参与活动状态计算 ✅

- **影响范围**: Flutter 客户端主入口、runtime 选择页、terminal 工作台、状态刷新策略

5. 初始化、切换 tab、创建/关闭 terminal 时，terminal 列表和数量均以后端快照为准。

## Session S012 - 状态语义拆解与后端权威在线态回流
> 时间: 2026-03-29 20:10
> 类型: bugfix
> 状态: confirmed
> 来源: 真实使用反馈
> **标记: 缺陷回流**

### 变更类型
- **类型**: 缺陷修复 + 状态模型重构
- **影响范围**: Server runtime 状态语义、Agent 在线稳定性、Flutter runtime 选择页与 terminal 页状态展示
- **紧急程度**: 高（已经影响 terminal 创建/关闭可用性与状态认知）

### 用户反馈（真实使用发现）
1. “电脑在线”的含义混乱，桌面端登录后仍可能看到设备离线。
2. 进入 terminal、退出再进入后，手机/电脑数量会显示为 0，表现不可信。
3. 关闭 terminal 后，再次创建 terminal 经常失败或得到不可用 terminal。
4. 用户确认正确语义应为：`电脑在线 = 当前电脑可以创建并承载被远程操作的 terminal`。

### 当前问题拆解
- `电脑在线`、`桌面客户端在线`、`terminal 在线`、`view 在线` 被混用，导致 UI 与行为理解错位。
- runtime `views` 的显示和关闭判定曾混用客户端本地缓存与 Redis 历史字段，缺少后端权威实时查询。
- 当前真实环境里出现 `agent_online=false + create_failed terminals`，说明 Agent 在线稳定性未闭环。

### 已确认结论
1. `电脑在线` 只等于 `device_agent_online=true`，也就是 **本机 Agent 在线且可承载 terminal**。
2. `桌面客户端打开` 不等于 `电脑在线`；它只是一个 view。
3. `mobile/desktop` 数量应由后端当前活跃连接实时计算，不应由客户端缓存或 Redis 脏聚合值主导。
4. terminal 页进入和列表页返回都应先向后端刷新当前状态，再由 ws presence 做增量更新。
5. 本轮需要新增补丁任务，不修改已完成多 terminal 主链路任务的核心定义。

### 规划结论
1. 新增“状态语义基线”任务，明确 `device / terminal / view` 三层在线语义和 UI 文案。
2. 新增“服务端权威状态”任务，统一 runtime API 对设备在线、terminal 实时视图数和关闭判定的来源。
3. 新增“Agent 在线稳定性”任务，收敛 `open -> close` 抖动与 `create_failed` 逃逸问题。
4. 新增“客户端状态对齐”任务，保证进入/返回页面时主动刷新后端状态，并统一桌面端本机在线文案。

### 待确认项
- [x] `电脑在线 = 本机 Agent 在线 = 可创建并承载 terminal` ✅
- [x] `mobile/desktop` 数量以后端实时状态为准 ✅
- [x] 客户端进入时先查后端，再接 ws 增量 ✅
- [x] 本轮通过新增补丁任务收口，不回写旧已完成任务定义 ✅

## Session S011 - 单 Agent 多 Terminal 架构升级规划
> 时间: 2026-03-29 01:40
> 类型: feature
> 状态: confirmed
> 来源: 架构演进与后续能力规划

### 变更类型
- **类型**: 架构升级 + 测试策略前置
- **影响范围**: Server 会话模型、Agent 运行时、Flutter 连接入口、`.dev-flow` 计划基线
- **紧急程度**: 高（决定后续多终端能力与异常恢复语义）

### 用户需求
1. 从“一个 session 对应一个终端”升级为“一个电脑一个 Agent，Agent 下多个 terminal”。
2. 每个 terminal 可以在不同目录下工作，并由客户端选择连接哪个 terminal。
3. 只有服务端确认电脑在线时，客户端才允许创建或连接 terminal。
4. 后续要支持“每台电脑最多多少个 terminal”的限制。
5. 优先把可 mock 的逻辑、状态机、协议、选择流程测试先做完，再做真实 PTY 和真机 smoke。

### 约束与结论
- 采用 **单设备单 Agent，多 terminal** 模型，不引入“一台电脑多个 Agent”作为当前范围。
- `terminal` 才是工作上下文载体，可独立拥有 `cwd / command / env / title`。
- 服务端状态需要从 `session` 升级为 `device + terminal + views` 三层。
- 客户端连接入口改为：先选在线设备，再选 terminal；设备离线时不允许进入 terminal 会话。
- 异常语义需要区分：`network_lost / agent_shutdown / terminal_exit / server_forced_close`。
- 采用“自动化优先”策略：先补契约、状态机、服务层、选择状态等 mock 测试，再进入真实 PTY/真机集成。

### 规划结论
1. 新增设备与 terminal 契约，明确设备在线、terminal 生命周期、连接资格和关闭原因。
2. 新增服务端设备注册、terminal 状态模型、在线 gating 和 terminal 列表 API 任务。
3. 新增 Agent 多 terminal runtime 任务，支持按 terminal 创建 PTY，并传入 `cwd / command / env`。
4. 新增断线原因与 grace period 任务，使“短线”和“主动关闭”在语义上可区分。
5. 新增客户端设备/terminal 选择状态与页面任务，并优先以 mock 测试覆盖选择、 gating、恢复和错误分支。
6. 将本轮测试策略明确为“先 mock/logic，再真实集成和手工 smoke”。

### 待确认项
- [x] 当前采用“单 Agent 多 Terminal”，不做“多 Agent”架构 ✅
- [x] terminal 支持不同 `cwd / command / env / title` ✅
- [x] 客户端必须先经过服务端在线校验，再能连接 terminal ✅
- [x] 本轮先尽可能完成可 mock 的逻辑和协议测试，再做真实集成 ✅

- **影响范围**: Flutter 客户端移动端终端交互、主题系统、`.dev-flow` 计划基线

## Session S018 - 桌面端 Agent 后台模式与退出语义
> 时间: 2026-03-29 09:20
> 类型: planning
> 状态: open
> 来源: 产品交互收敛

### 变更类型
- **类型**: 规划增量
- **影响范围**: 桌面端、Agent、本地配置、设备在线语义
- **紧急程度**: 高（持续影响“电脑在线”与终端创建理解）

### 用户新增需求
1. 采用 **方案 B**：Agent 作为本机独立后台服务，桌面端是本机控制台，不再把桌面端当成普通 remote client。
2. 需要一个开关控制：**关闭桌面端后，是否让 Agent 继续后台运行**。
3. `电脑在线` 的产品语义继续保持为：**本机 Agent 在线，当前电脑可创建并承载 terminal**。
4. 桌面端退出后，如果选择“不后台运行”，则应让设备很快从在线收口为离线。

### 当前逻辑漏洞
1. **桌面端与 Agent 生命周期没有明确所有权**
   - 关闭桌面端后，Agent 可能继续存活，用户会误以为“电脑明明关了怎么还在线”。
2. **缺少后台模式配置**
   - 现在无法表达“后台继续在线”还是“退出即离线”的用户偏好。
3. **缺少 Agent 所有权语义**
   - 桌面端若不区分“自己拉起的 Agent”和“外部已存在的 Agent”，退出时可能误杀不该停止的 Agent。
4. **退出顺序可能导致假在线**
   - 如果桌面端先退出，Agent 再被动超时，手机端会在 60 秒窗口内继续看到在线。
5. **重复启动风险**
   - 桌面端如果每次启动都盲目拉起 Agent，可能出现多个本地 Agent 竞争同一 device/session。

### AI 理解
需要把桌面端从“普通 client”升级成“本机 Agent 控制台”：
- 桌面端负责探测本机 Agent 是否在线
- 可按配置决定在桌面端退出时是否停止 Agent
- 桌面端只在“自己拉起的 Agent”且用户关闭后台模式时，才在退出时停止 Agent
- Agent 在线仍然是服务端唯一权威状态，不由桌面端 UI 是否打开决定

### 风险点
- `startup`: 桌面端冷启动时，Agent 启动与登录恢复顺序可能导致首页抖动
- `config`: 后台运行开关需要本地持久化，升级时要有默认值与兼容逻辑
- `first_use`: 首次启动桌面端时，用户需要明确理解“后台继续在线”的含义
- `network`: Agent 优雅下线失败时，服务端可能短时间仍显示在线
- `auth`: Agent 重启后 token 失效/刷新失败会直接影响“后台继续在线”体验

## Session S021 - Agent 终端管理系统性重构与后台运行模式
> 时间: 2026-03-30 10:00
> 类型: planning
> 状态: confirmed
> 来源: 架构重构与系统性问题解决

### 变更类型
- **类型**: 架构重构 + 生命周期管理
- **影响范围**: Server、Agent、桌面端 Flutter、手机端 Flutter
- **紧急程度**: 高（当前"关闭终端后无法创建"是系统性问题）

### 核心问题
1. **Agent 生命周期与 Terminal 生命周期耦合**
   - WebSocket 断开时，Server 端 `_cleanup_agent` 立即清理 Agent 连接
   - 但 Agent 进程可能仍在运行，导致状态不一致

2. **桌面端 vs 手机端行为差异未明确区分**
   - 桌面端：Agent 运行在本地，是"主机"
   - 手机端：Agent 运行在远程设备，只是"查看器"
   - 两者生命周期管理模式应该完全不同

3. **后台运行模式需要系统设计**
   - 需要一个开关：`keepAgentRunningInBackground`
   - 桌面端关闭时，根据开关决定是否停止 Agent
   - 需要区分"自己启动的 Agent"和"外部已存在的 Agent"

4. **跨平台状态文件权限问题**
   - 状态文件路径需要考虑 macOS 沙盒、Windows、Linux 差异
   - 需要使用平台标准目录

5. **端口发现与冲突处理**
   - Agent 本地 HTTP Server 需要固定端口范围
   - 需要处理端口冲突、孤儿进程检测

### 已确认结论

#### 1. 终端数据流保持不变
```
Agent PTY → WebSocket → Server → WebSocket → Clients (桌面/手机)
```
- 保持 Agent → Server → Client 的数据流
- 原因：多端同步、历史记录、权限控制、离线查看

#### 2. 桌面端与手机端行为模式区分
- **桌面端**：
  - Agent 运行在本地
  - 有"后台运行"开关
  - 需要管理 Agent 生命周期
  - 通过本地 HTTP 与 Agent 通信

- **手机端**：
  - Agent 运行在远程设备
  - 只是查看器/控制器
  - 不管理 Agent 生命周期
  - 没有"后台运行"概念

#### 3. 本地 HTTP Supervisor 方案
- Agent 启动时同时启动本地 HTTP Server（端口 18765-18769）
- 提供 `/health`, `/status`, `/stop`, `/config`, `/terminals` 端点
- Flutter UI 通过 HTTP 与 Agent 通信
- 状态文件记录实际端口和 PID

#### 4. 状态文件路径策略
- 使用平台标准目录：

## Session S022 - PTY 进程组清理与资源完整性
> 时间: 2026-03-30 15:00
> 类型: bugfix
> 状态: confirmed
> 来源: 代码审查与真实使用风险分析
> **标记: 缺陷回流**

### 变更类型
- **类型**: 缺陷修复 + 资源管理增强
- **影响范围**: Agent PTY 进程管理、进程组清理、资源释放
- **紧急程度**: 高（可能导致孤儿进程、僵尸进程、资源泄漏）

### 核心问题
1. **只杀死直接子进程，不杀进程组**
   - 当前 `PTYWrapper.stop()` 使用 `os.kill(self.pid, signal.SIGTERM)`
   - 如果子进程 fork 了孙子进程（如 shell → bash 脚本 → 其他命令），孙子进程会变成孤儿进程
   - `os.setsid()` 已在子进程中创建新会话，但父进程停止时没有用 `os.killpg()` 杀死整个进程组

2. **没有超时后的 SIGKILL 强制终止**
   - `os.waitpid(self.pid, 0)` 会无限等待
   - 如果子进程不响应 SIGTERM，会一直阻塞

3. **资源清理可能不完整**
   - 文件描述符关闭后，异步读取任务可能仍在运行
   - 内存中可能残留未清理的缓冲区

4. **缺少进程组清理的单元测试**
   - 当前测试没有覆盖进程组清理场景
   - 没有测试超时后的强制终止逻辑

### 已确认结论
1. **进程组清理方案**：
   - 在 `_child_process()` 中使用 `os.setsid()` 创建新会话（已实现）
   - 在 `stop()` 中使用 `os.killpg()` 杀死整个进程组，而不是只杀死直接子进程
   - 使用 `-self.pid` 作为进程组 ID（因为进程组 leader 的 PGID 等于其 PID）

2. **超时 + SIGKILL 方案**：
   - 发送 SIGTERM 后，使用 `select` 或 `asyncio.wait_for` 设置超时
   - 超时后发送 SIGKILL 强制终止
   - 推荐超时时间：3 秒

3. **资源清理方案**：
   - 确保异步读取任务在关闭 fd 前取消
   - 使用 try-finally 确保资源释放
   - 添加显式的资源状态检查

4. **测试覆盖方案**：
   - 单元测试：进程组清理、超时终止、资源释放
   - 集成测试：真实 fork 场景、孤儿进程检测

### 风险点
- `startup`: Agent 启动时 PTY 创建失败需要确保不残留资源
- `first_use`: 首次使用时如果进程组清理失败，可能影响系统稳定性
- `network`: 网络断开触发终端关闭时，需要确保进程被正确清理

### 待确认项
- [x] 使用 `os.killpg()` 杀死整个进程组 ✅
- [x] 添加 3 秒超时后的 SIGKILL 强制终止 ✅
- [x] 确保异步任务在 fd 关闭前取消 ✅
- [x] 添加进程组清理的单元测试和集成测试 ✅

## Session S020 - 关闭最后一个 terminal 后的状态归一化
> 时间: 2026-03-29 12:10
> 类型: planning
> 状态: open
> 来源: 缺陷回流

### 变更类型
- **类型**: 缺陷回流
- **影响范围**: 桌面工作台状态机、terminal 生命周期收口、首个 terminal 创建体验
- **紧急程度**: 高（用户关闭最后一个 terminal 后仍看到“启动本机 Agent 失败”）

### 用户新增问题
1. 关闭唯一 terminal 后，桌面端工作台仍停留在历史 `createFailed` 态。
2. 该状态与“当前已经回到空工作台，应该允许再次创建第一个 terminal”的语义冲突。
3. 用户明确要求继续按系统性方式解决，不接受临时补丁。

### 当前问题判断
1. `createFailed` 仍被当作持久失败标记，而不是“本次 bootstrap 尝试的结果”。
2. terminal 生命周期变化（尤其是关闭最后一个 terminal）后，没有触发工作台状态的归一化。
3. 这会让空工作台继续显示历史失败页，掩盖真实的当前状态。

### AI 理解
需要把 `bootstrapAttemptState` 与 terminal 集合变化解耦：
- `createFailed` 只能绑定一次具体 bootstrap 尝试
- 当最后一个可用 terminal 被关闭后，工作台必须重新归一化到：
  - `readyToCreateFirstTerminal`
  - 或 `bootstrappingAgent`
  - 或重新计算后的 `createFailed`
- 不能沿用历史失败结果直接卡死页面

### 风险点
- `first_use`: 关闭最后一个 terminal 后空工作台是否可恢复到正确的首次创建路径
- `startup`: bootstrap 失败标志若不及时清理，会持续污染桌面端首次启动体验
- `state_machine`: 关闭 terminal、自动 bootstrap、手动重试三个事件若顺序不清，会再次引入页面语义冲突
