# Session S042: 设备感知智能终端进入长期路线补全

> 日期：2026-04-22
> 状态：已规划，待执行

## 需求来源

用户确认：

- 项目还未正式上线，希望一次性把智能终端进入相关需求全部规划完整
- 不能把“智能识别”做成只适配开发者自己电脑的规则
- 后续如果接入 LLM，必须以“当前用户、当前设备、当前项目事实”为基础，而不是凭空猜目录

## 本轮规划结论

### 1. 智能识别分三层，不直接跳黑盒

- **v1**：本地可解释推荐 + 轻量短句规则
- **v2**：引入“设备级项目上下文”候选源
- **v3**：在候选源之上接入可选 LLM Planner

LLM 不是第一层事实源，只能在事实之上做选择和补全。

### 2. 路径真相必须来自当前设备

后续智能识别只能使用以下来源：

- 当前设备最近成功进入过的 `cwd`
- 当前设备已有 terminal 历史
- 用户显式固定/收藏的项目
- Agent 在“用户已授权范围内”发现的项目候选
- 用户本次输入里显式写出的路径

禁止让模型自行发明一个设备上从未出现过的本地路径。

### 3. Server/Agent/Client 的职责保持稳定

- **Agent**：采集设备本地项目候选摘要，不解析自然语言
- **Server**：只中转/缓存候选摘要，不持有完整文件树，不做 LLM 推理
- **Client**：负责推荐、意图解析、候选展示、确认流和最终 `TerminalLaunchPlan`

### 4. LLM 只能输出受约束的结果

后续若接入 LLM，输出必须满足：

- 仍然归一化为 `TerminalLaunchPlan`
- 必须引用某个 `ProjectContextCandidate` 或用户显式输入路径
- 不在候选集中的路径默认 `requires_manual_confirmation=true`
- provider 不可用时必须回退到本地规则，不阻断创建

## 新增规划范围

新增一条长期路线：

1. Agent/Server 提供设备级 `ProjectContextCandidate`
2. Client 按设备缓存与排序候选项目上下文
3. Client 增加 `PlannerProvider` 抽象：`local_rules` / `llm`
4. UI 增加“候选项目展示 + 需确认分支”
5. 补齐隐私、跨设备污染、fallback 和真实首用测试

## 风险与约束

- `privacy`：不得把完整本地文件树上传到 Server
- `config`：模型不能生成脱离设备事实的路径
- `llm`：provider 不可用、超时、空结果都必须回退
- `first_use`：首次使用时即使没有候选项目，也必须仍可创建 shell / Claude / Codex
