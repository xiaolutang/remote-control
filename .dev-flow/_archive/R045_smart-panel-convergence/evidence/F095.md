## 测试场景

### 1. 活跃轮次编辑第一轮回答
- 两轮问答后 result 状态 → 点击第一个回答进入 inline edit → 提交
- 验证 `runSession` 被调用，`truncateAfterIndex=1`（截断到第一个 question 之后）
- 验证重跑结果可见

### 2. 活跃轮次编辑第二轮回答保留第一轮
- 两轮问答（AnsAlpha, AnsBeta）→ 点击第二个回答编辑
- 验证 `truncateAfterIndex=3`（保留第一轮 Q&A，截断到第二个 question 之后）
- 验证 `runSession` 被调用

### 3. 历史轮次编辑回答移到活跃
- 建立历史轮次 → 点击历史中的回答进入 inline edit
- 验证 `truncateAfterIndex=1`（截断到历史 intent 的 question 之后）
- 验证 `runCount=2`，最后一次 runIntent 是历史意图

### 4. 单轮问答编辑（truncates all Q&A and reruns）
- 一轮问答 → 编辑该回答
- 验证 `truncateAfterIndex=1`，runSession 被调用

### 5. 空 server events 时编辑
- 纯 SSE 建立问答（`_serverConversationEvents` 为空）→ 编辑回答
- 验证 `_truncateConversationEventsForAnswer` early return 不崩溃
- 验证 `truncateAfterIndex=-1`，runSession 被调用

## 已知限制

**`newAnswer` 参数未消费**：`_submitAnswerEdit` 的 `newAnswer` 参数在当前实现中未被使用。编辑回答的流程是"截断事件 → 重跑原始 intent → agent 重新提问 → 用户通过正常 respond 提供新回答"。这是一个设计选择而非 bug，但 `newAnswer` 参数在 API 签名中是多余的。如果未来需要将编辑后的回答直接作为上下文传入重跑，需修改实现。此问题不在 F095（纯测试任务）范围内。

- `_submitInlineEdit()`: 输入新文本并提交 inline edit 的 helper
- `runTruncateAfterIndexes`: 追踪 `runSession` 的 `truncateAfterIndex` 参数
